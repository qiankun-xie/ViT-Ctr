## Recommended Architecture

The system follows the same three-layer architecture as ViT-RR, adapted for RAFT chemistry. All three layers are independent enough to build and test separately, which makes phased development straightforward.

## Component Boundaries

| Component | File(s) | Responsibility | Communicates With |
|-----------|---------|---------------|-------------------|
| ODE Solver | `raft_ode.py` | Integrate RAFT moment equations (Mn, Đ vs. conversion) given kinetic parameters | param_sampler, dataset_builder |
| Parameter Sampler | `param_sampler.py` | Sample parameter space: log10(Ctr), kadd/kfrag, [CTA]/[M], initiator loading, RAFT type | raft_ode, dataset_builder |
| ctFP Encoder | `ctfp_encoder.py` | Convert (conversion, Mn, Đ) trajectory → 64×64×2 image tensor; shared by dataset_builder and deploy.py | dataset_builder, deploy.py |
| Dataset Builder | `dataset_builder.py` | Orchestrate parallel ODE runs, collect ctFPs + labels, save to .pt file | raft_ode, param_sampler, ctfp_encoder |
| SimpViT Model | `model.py` | 2-channel 64×64 patch-based Transformer; output dim=3 (log10(Ctr), inhibition, retardation) | train.py, deploy.py |
| Training Script | `train.py` | DataLoader, loss, optimizer, checkpoint saving | model.py, dataset |
| Evaluation Module | `evaluate.py` | Test-set metrics, parity plots, literature validation, bootstrap CI computation | model.py, dataset |
| Web App | `deploy.py` | Streamlit UI: data input, ctFP encoding, model inference, bootstrap UQ display | ctfp_encoder.py, model.py |
| Paper / SI | `paper/`, `si/` | LaTeX/Word documents; figures generated by evaluate.py | evaluate.py outputs |

### Key Boundary Rule
`ctfp_encoder.py` must be a standalone importable module with no Streamlit or training dependencies. Both the dataset builder and the web app call it identically. This is the critical shared interface — any change to encoding logic must propagate to both paths.

## Data Flow

```
RAFT kinetic parameters
(Ctr, kadd, kfrag, [CTA]/[M], kp, kt, f, kd)
         │
         ▼
   scipy.integrate.solve_ivp (Radau, stiff)
   moment equations → Mn(α), Đ(α) trajectory
         │
         ▼
   ctfp_encoder.transform()
   x-axis: [CTA]/[M]  (0→1 → pixel 0→63)
   y-axis: conversion  (0→1 → pixel 0→63)
   Ch0: Mn values at each (x, y) point
   Ch1: Đ values at each (x, y) point
   Output: torch.Tensor shape (2, 64, 64)
         │
         ▼
   SimpViT.forward(x)
   patch_embedding (Conv2d 2→64, kernel=16)
   → 16 patches → positional embedding
   → 2× TransformerEncoderLayer (d=64, heads=4)
   → mean pool over patches (no CLS token)
   → fc Linear(64, 3)
   Output: [log10(Ctr), inhibition_period, retardation_factor]
         │
         ▼
   Bootstrap (200 iterations of input noise)
   → covariance matrix → F-distribution 95% JCI
   Output: mean ± half-width for each of 3 parameters
```

### Inverse Transform (Training Labels)
Labels are stored as log10 of physical values (matching ViT-RR convention). During inference, outputs are exponentiated: `10^pred → physical value`. This compression keeps regression targets in a well-conditioned range.

## Patterns to Follow

### Pattern 1: Shared Encoder (Verified from ViT-RR codebase)
**What:** The `transform()` function that converts raw experimental data to image tensor is defined once and imported by both the dataset builder and the Streamlit app.
**When:** Always — encoding consistency between training data and inference is non-negotiable.
**Example (ctfp_encoder.py):**
```python
import numpy as np, math, torch

def transform(data, img_size=64):
    """
    data: list of (cta_ratio, conversion, Mn, D) rows
    Returns: torch.Tensor shape (2, img_size, img_size)
    Ch0 = Mn, Ch1 = D (dispersity)
    x-axis = [CTA]/[M] fraction, y-axis = conversion
    """
    img = np.zeros((2, img_size, img_size))
    for cta_ratio, conv, mn, dispersity in data:
        col = min(math.floor(cta_ratio * img_size), img_size - 1)
        row = min(math.floor(conv * img_size), img_size - 1)
        img[0][row][col] = mn        # channel 0: Mn
        img[1][row][col] = dispersity  # channel 1: D
    return torch.tensor(img, dtype=torch.float32)
```

### Pattern 2: ODE Moment Equations for Mn and Đ
**What:** Use scipy `solve_ivp` with `method='Radau'` (implicit, handles stiff RAFT systems) and the method of moments to compute zeroth/first/second moments of the chain length distribution. RAFT kinetics produce stiff ODEs due to fast RAFT equilibrium vs. slow propagation.
**When:** Always in the data generation layer. Never re-derive Mn/Đ via other means for the training set.
**Key parameters to track:** [M], [CTA], [I], and moments λ0, λ1, λ2 (dead chains) + μ0, μ1, μ2 (living chains). Sample points at equal conversion intervals (not time intervals) for consistent fingerprint density.

### Pattern 3: Bootstrap Uncertainty by Input Perturbation
**What:** For inference, run the model 200 times on noise-perturbed versions of the input (Gaussian noise at 3% of each value). Collect predictions, compute covariance, apply F-distribution CI at 95%. Matches ViT-RR exactly — verified from `predict_model()` in `deploy.py`.
**When:** Always in the web app. Optionally skip during development iteration (checkbox toggle).

### Pattern 4: Log10 Label Space
**What:** Store Ctr as log10(Ctr) in training labels. The model predicts log10 space; convert to linear for display.
**When:** Always — Ctr spans several orders of magnitude across RAFT agent types. Linear regression on raw Ctr values would produce poor gradients.

### Pattern 5: Parameter Space Coverage by RAFT Agent Type
**What:** Stratify synthetic data generation by RAFT agent class (dithioester, trithiocarbonate, xanthate, dithiocarbamate). Each class has a characteristic Ctr range. Sample uniformly within each class's physical range to avoid imbalanced training.
**When:** During dataset_builder design. Failure to stratify results in poor predictions outside the most common class.

## Anti-Patterns to Avoid

### Anti-Pattern 1: Separate Encoding Logic in App vs. Training
**What:** Reimplementing the ctFP transform inside `deploy.py` independently from the training-time encoder.
**Why bad:** Any subtle difference (axis order, normalization, edge handling) between training and inference encoding invalidates predictions. This is a silent failure — the model will produce plausible but wrong numbers.
**Instead:** Import from the single shared `ctfp_encoder.py` module everywhere.

### Anti-Pattern 2: Sampling at Equal Time Intervals
**What:** Running the ODE to fixed time points and using those as fingerprint entries.
**Why bad:** Early time points cluster at low conversion; the fingerprint image is nearly empty at high conversion. The ctFP encodes data density by (CTA ratio, conversion) position — uniform time steps waste pixel resolution.
**Instead:** Interpolate the ODE solution to a fixed grid of conversion values (e.g., 0.02, 0.04, ..., 0.98) using `dense_output=True` from `solve_ivp`.

### Anti-Pattern 3: Training Directly on Physical Mn Values
**What:** Using raw Mn (in g/mol, typical range 1,000–200,000) as channel values.
**Why bad:** Extreme value range dominates pixel intensity. The model learns large-Mn systems and ignores small ones.
**Instead:** Normalize Mn by theoretical Mn at full conversion (Mn_theory = [M]0 / [CTA]0 × MW_monomer) to get a dimensionless ratio near 1 for well-controlled RAFT.

### Anti-Pattern 4: Monolithic deploy.py
**What:** All logic (encoding, model loading, bootstrap, UI) in one file like ViT-RR's deploy.py.
**Why acceptable in ViT-RR (2 models, 2 chemistry types) but problematic here:** ViT-Ctr adds RAFT-type-specific parameter ranges and the potential Route A (structure → Ctr) path. A monolithic file becomes hard to test.
**Instead:** Extract `ctfp_encoder.py` and `model.py` as importable modules from the start, even if the app remains a single Streamlit script.

### Anti-Pattern 5: Mixing Route A and Route B in the Same Model
**What:** Training a single SimpViT that takes both ctFP (Route B input) and molecular structure descriptors (Route A input).
**Why bad:** The two routes have completely different data requirements, architectures, and validation strategies. Coupling them delays both.
**Instead:** Build Route B first as the primary deliverable. Route A is an independent, separate model added in a final exploratory phase.

## Component Build Order (Phase Dependencies)

The dependency graph determines build order. Each component depends only on components above it.

```
Phase 1: ODE Foundation
  └── raft_ode.py (standalone, no ML dependencies)
  └── param_sampler.py (no ML dependencies)
  └── ctfp_encoder.py (no ML, no ODE — pure image encoding)

Phase 2: Dataset Generation
  └── dataset_builder.py → depends on raft_ode + param_sampler + ctfp_encoder
  └── Produces: dataset.pt (ctFPs + labels)

Phase 3: Model Training
  └── model.py (SimpViT, output_dim=3) → standalone
  └── train.py → depends on model + dataset.pt

Phase 4: Evaluation & Validation
  └── evaluate.py → depends on model.py + trained .pth + literature data
  └── Produces: figures, metrics, parity plots (inputs to paper)

Phase 5: Web Deployment
  └── deploy.py (Streamlit) → depends on ctfp_encoder + model.py + .pth
  └── No dependency on dataset.pt or train.py

Phase 6: Paper & SI Writing
  └── Depends on all evaluate.py outputs
  └── Route A exploration runs in parallel with paper writing
```

**Critical path:** ODE correctness (Phase 1) directly determines dataset quality (Phase 2), which determines model quality (Phase 3). Errors discovered in Phase 4 may require regenerating the dataset. Budget time for one ODE validation iteration before large-scale data generation.

## Scalability Considerations

| Concern | Local (i5-10210U + MX350) | Colab Free GPU |
|---------|--------------------------|----------------|
| ODE simulation speed | ~1,000 param sets/min single-core; use multiprocessing.Pool | Not needed for ODE |
| Dataset size | 1M samples × 64×64×2 float32 = ~32 GB uncompressed; use chunked .pt files or HDF5 | Load via Google Drive mount |
| Model training | SimpViT ~3.4MB; 1M samples = slow on CPU. Use Colab for training runs | ~1-2 hrs per training run |
| Inference (deploy) | CPU inference, <100ms per prediction + bootstrap adds ~5s for 200 iters | N/A — Streamlit Community Cloud (CPU) |
| Memory during training | Batch size 256 on MX350 2GB VRAM: feasible for 64×64×2 float32 images | Batch size 512+ on Colab T4 |

**Recommended dataset storage strategy:** Generate in chunks of 100K samples, store as `dataset_chunk_000.pt` through `dataset_chunk_009.pt`. During training, use `ConcatDataset` or streaming to avoid loading all 32GB into RAM at once.
